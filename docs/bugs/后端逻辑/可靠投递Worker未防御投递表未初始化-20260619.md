# 可靠投递 Worker 未防御投递表未初始化

## 根因分析

- 涉及后端 Worker：`backend/src/main/java/io/litealert/notify/delivery/NotifyDeliveryWorker.java:tick()`。
- 涉及后端 Store：`backend/src/main/java/io/litealert/notify/delivery/NotifyDeliveryStore.java:recoverStuck()`。
- 涉及数据库表：`la_notify_delivery`。
- 数据流转链路：应用启动 → Spring Scheduling 触发可靠投递 Worker → Worker 执行 `recoverStuck()` → JDBC 执行 `update la_notify_delivery ...` → 当前数据库中尚无 `la_notify_delivery` → H2 抛 `Table "LA_NOTIFY_DELIVERY" not found`。
- 根本原因：可靠投递 Worker 默认投递表一定存在，没有在执行 SQL 前检查表是否准备好。虽然 Flyway 空库初始化测试可以创建 V1/V2 表结构，但实际启动中可能存在 IDE 编译产物未刷新、Flyway 未启用或 Worker 早于表可见状态执行的情况，Worker 缺少防御导致启动失败。

## 解决方案

- `NotifyDeliveryStore` 增加 `tableReady()`，检查 `la_notify_delivery` 是否存在。
- `NotifyDeliveryWorker.tick()` 开始时先调用 `tableReady()`。
- 如果表不存在，本轮 Worker 直接跳过并记录 warn，不执行 `recoverStuck()` 或扫描 SQL。
- 保留 Flyway 作为正式建表机制；Worker 防御只用于避免启动期因表未准备好导致应用失败。

# 可靠投递表缺失导致 Worker 启动失败

## 根因分析

- 涉及后端 Worker：`backend/src/main/java/io/litealert/notify/delivery/NotifyDeliveryWorker.java:tick()`。
- 涉及后端 Store：`backend/src/main/java/io/litealert/notify/delivery/NotifyDeliveryStore.java:recoverStuck()`。
- 涉及数据库迁移脚本：`backend/src/main/resources/db/migration/h2/V2__notify_delivery.sql`。
- 数据流转链路：应用启动 → Spring Scheduling 启动可靠投递 Worker → Worker 执行 `recoverStuck()` 恢复超时 `SENDING` 任务 → JDBC 执行 `update la_notify_delivery ...` → 旧 H2 数据库中缺少 `la_notify_delivery` 表 → H2 抛出 `JdbcSQLSyntaxErrorException`。
- 根本原因：可靠投递 Worker 默认认为 `la_notify_delivery` 表一定存在，但升级场景中可能出现 Flyway baseline / migration 状态与实际表结构不一致，或者使用旧库启动时 V2 没有实际补表；代码缺少“表不存在时安全跳过”的防御，migration 也缺少后续幂等兜底脚本。

## 解决方案

1. 新增 V3 幂等兜底 migration：
   - 对 H2 / PostgreSQL / GaussDB 使用 `create table if not exists la_notify_delivery` 和 `create index if not exists`。
   - 对 MySQL / OceanBase 提供同等建表脚本，用于未执行 V2 的升级场景。
2. `NotifyDeliveryStore` 增加 `tableReady()` 检查，通过 JDBC metadata / 查询方式判断 `la_notify_delivery` 是否存在。
3. `NotifyDeliveryWorker.tick()` 在表不存在时跳过本轮扫描并记录 warn，避免启动后持续抛异常。
4. 增加测试覆盖：旧库缺表时 Worker tick 不抛异常；迁移脚本补表后正常运行。

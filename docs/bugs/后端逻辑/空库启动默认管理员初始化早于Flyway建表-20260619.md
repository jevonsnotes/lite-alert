# 空库启动时默认管理员初始化早于 Flyway 建表

## 根因分析

- 涉及后端初始化类：`backend/src/main/java/io/litealert/common/db/DatabaseInitializer.java:run()`。
- 涉及数据库迁移：`backend/src/main/resources/db/migration/h2/V1__baseline.sql`。
- 数据流转链路：应用启动 → Flyway 应创建基础表 → `DatabaseInitializer.run()` 初始化默认管理员 → 查询 `la_user` → 空库中 `la_user` 尚未创建 → H2 抛 `Table "LA_USER" not found`。
- 根本原因：引入 Flyway 后，`DatabaseInitializer` 不再负责建表，但没有显式声明必须在 Flyway 初始化完成之后再执行，导致空库场景下存在初始化顺序风险。

## 解决方案

- 在 `DatabaseInitializer` 上声明依赖 Spring Boot 的 `flywayInitializer`，确保 Flyway migration 完成后再执行默认管理员初始化。
- 保持 `DatabaseInitializer` 只做基础数据 bootstrap，不重新接管 schema 建表。
- 使用空库启动相关后端测试验证 Flyway 能先创建 `la_user`，再初始化默认管理员。

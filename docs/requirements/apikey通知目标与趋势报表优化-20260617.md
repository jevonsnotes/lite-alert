# API key、通知目标与仪表盘趋势优化需求分析

**日期**：2026-06-17
**类型**：功能优化
**关联模块**：ApiKey 管理、通知目标管理、管理员仪表盘

## 1. 需求概述

本次优化主要解决三个使用体验问题：

1. **ApiKey 管理更可维护**：用户已经创建的 ApiKey 需要能修改名称、授权范围、有效期等元信息；完整 key 原文仍然不支持查看，但需要提供“轮换/重新生成 Key”能力，用户丢失 Key 时生成新 Key 并替换旧 Key。同时列表需要支持搜索，方便 key 较多时定位。
2. **通知目标管理更易用**：通知目标需要支持编辑和搜索；添加/编辑弹窗中的类型选择改成下拉框，并加宽弹窗，避免表单标签过短导致换行影响观感。
3. **仪表盘趋势更细**：管理员仪表盘除整体趋势外，需要能按 Topic 和 ApiKey 维度查看趋势，便于判断“哪个 Topic 调用多、哪个 Key 触发多或失败多”。

## 2. 技术可行性结论

结论：**可行，且不需要引入数据库或新的外部服务。**

原因：

- 后端已有 `PATCH /api/apikeys/{id}`，`ApiKeyService.update()` 已支持修改 `name / scopes / validUntil`，且实体中 `keyHash` 不会返回给前端。前端目前只是没有提供“编辑”入口。
- 后端已有 `PATCH /api/contacts/{id}`，`NotifyTargetService.update()` 已支持修改通知目标的 `label / enabled / endpoint / secret`。前端目前只使用它做启用/禁用，没有提供完整编辑弹窗。
- 审计日志 `webhook.accepted` 已记录 `topicId`、API_KEY 模式下已记录 `apiKeyId`。现有 `StatsController.daily()` 已经能扫描 audit 日志生成整体趋势，因此可以扩展一个带维度参数的趋势接口。
- 前端已引入 ECharts，仪表盘已有整体折线图，新增 Topic / Key 维度图表可以复用同一套渲染方式。

## 3. 推荐方案及理由

### 3.1 ApiKey 管理

推荐方案：前端增强为“可搜索 + 可编辑元信息 + 可轮换 Key”。

- 列表顶部增加搜索框，支持按以下内容本地过滤：
  - 名称
  - key 前缀
  - 状态
  - 授权范围显示名（命名空间名、Topic 名）
- 表格操作列增加“编辑”和“轮换”按钮。
- 编辑弹窗复用新建弹窗的主要字段：
  - 名称
  - 有效期：永久有效 / 指定失效时间
  - 授权范围：按命名空间、按 Topic 多选
- 编辑弹窗中只展示 `prefix`，并提示“完整 Key 不可查看；如已丢失，请使用轮换生成新 Key”。
- 保存时调用已有 `PATCH /api/apikeys/{id}`：
  - 永久有效时传 `clearValidUntil: true`
  - 指定失效时间时传 `validUntil`
  - scopes 为空时前端拦截并提示
- key 原文不进入编辑表单，不提供二次查看入口。
- 轮换时新增后端接口，建议为 `POST /api/apikeys/{id}/rotate`：
  - 校验当前用户有权操作该 Key。
  - 保留原 Key 的名称、授权范围、有效期等元信息。
  - 重新生成完整 key、prefix 和 keyHash。
  - 旧完整 key 立即失效。
  - 响应中只在本次轮换完成后返回一次新的完整 key，前端弹窗提醒用户立即复制保存。
  - 写入 `apikey.rotate` 审计日志。

理由：符合 `docs/design/11-apikey.md` 中“服务端不存原文 key，只存哈希；丢失只能重发，无法找回”的安全设计，同时满足用户丢失 Key 后的恢复使用诉求。

### 3.2 通知目标管理

推荐方案：前端补齐编辑能力，同时后端小幅增强审计与重复校验。

- 列表顶部增加搜索框，支持按类型、名称、endpoint 文本过滤。
- 操作列增加“编辑”按钮。
- 添加和编辑使用同一个弹窗：
  - 弹窗宽度从当前 `540px` 调整到约 `720px`。
  - `label-width` 调整到约 `120px`，避免“Webhook URL”等标签换行。
  - 类型选择从 radio button 改为 `el-select` 下拉框。
  - 新增时允许选择类型；编辑时建议不修改类型，避免“邮件地址改成机器人 URL”这种跨类型迁移带来歧义。如后续确需支持类型变更，可新增后端 type 更新校验。
  - secret 字段编辑时提示：留空表示清除/不设置；如需要“不修改现有密钥”，可增加“保持原密钥”交互。
- 后端 `NotifyTargetService.update()` 建议补充 `target.update` 审计日志，并在 endpoint 变更时复用创建时的重复 endpoint 校验。

理由：现有接口已经支持主体功能，主要是前端交互缺口；后端补审计有助于后续追踪谁修改了通知目标。

### 3.3 仪表盘 Topic / Key 维度趋势

推荐方案：扩展统计接口，前端增加维度切换和筛选。

后端：

- 在 `StatsController` 增加维度趋势接口或扩展现有 `/api/admin/stats/daily`：
  - 参数示例：`dimension=OVERALL|TOPIC|APIKEY`
  - 可选参数：`topicId`、`apiKeyId`
- 返回结构建议保持与现有图表兼容：
  - `labels`
  - `accepted`
  - `sent`
  - `failed`
  - `dimension`
  - `seriesLabel`
- 统计来源仍为 audit 日志：
  - `webhook.accepted` 计入受理。
  - `notify.sent` 计入送达。
  - `notify.failed` / `notify.give_up` 计入失败。
  - Topic 维度使用事件中的 `topicId`。
  - ApiKey 维度使用事件中的 `apiKeyId`；公开 Topic（NONE 模式）没有 key，按“不适用”处理或仅在选择 APIKEY 维度时排除。

前端：

- 仪表盘趋势卡片增加维度选择：整体 / Topic / ApiKey。
- 选择 Topic 维度时显示 Topic 下拉筛选，选项来自 `/topics`。
- 选择 ApiKey 维度时显示 ApiKey 下拉筛选，选项来自 `/apikeys`。
- 图表标题随筛选变更，例如：
  - `通知趋势 - 整体`
  - `通知趋势 - Topic：订单/支付成功`
  - `通知趋势 - ApiKey：订单系统-生产`

理由：最小改动复用已有审计和 ECharts 能力，符合项目“轻量、无数据库”的定位。

## 4. 风险点和注意事项

1. **ApiKey 安全边界不能破坏**
   - 不能查看或保存旧的完整 key 原文。
   - 编辑只允许改元信息：名称、有效期、授权范围。
   - 丢失 key 时通过“轮换/重新生成”获得新 key，旧 key 立即失效。

2. **通知目标 secret 编辑语义要清楚**
   - 当前后端 `secret: ""` 会清空密钥。
   - 如果编辑弹窗默认把 secret 留空，可能造成用户误以为“保持原密钥”，实际却清空。因此需要在前端明确提示，或设计为“留空不修改，点击清除才清空”。该点开发时需要重点处理。

3. **趋势图统计依赖 audit 日志完整性**
   - 历史日志如果缺少 `apiKeyId`，ApiKey 维度无法回补。
   - NONE 模式公开 Topic 没有 ApiKey，ApiKey 维度统计时应排除或显示为“无 ApiKey”。

4. **大范围日志扫描性能**
   - 当前 `StatsController` 已限制最多 365 个日点。继续沿用这个限制即可。
   - 若后续 audit 日志每天非常大，再考虑做每日统计快照；本次不引入。

## 5. 预估工作量

粗略估计：**1.5 - 2.5 个工作日**。

- ApiKey 前端编辑、搜索与轮换交互：0.7 天
- 通知目标前端编辑、搜索、弹窗优化：0.5 天
- 后端补测试、通知目标更新审计/重复校验：0.3 天
- 仪表盘维度趋势接口与前端图表筛选：0.7 - 1 天
- 联调、类型检查、后端测试、构建验证：0.3 - 0.5 天

## 6. 初步测试方案

遵循 TDD，先补测试再实现：

1. 后端 ApiKey：验证 update 不会改变 `prefix/keyHash`，且能更新名称、有效期、scope；验证 rotate 会生成新的 `prefix/keyHash` 并只返回一次新的完整 key。
2. 后端通知目标：验证 update 能修改 label、endpoint、enabled、secret；endpoint 重复时拒绝；审计记录写入。
3. 后端趋势统计：构造包含 `topicId/apiKeyId` 的 audit 日志，验证按 Topic / ApiKey 过滤后的 accepted/sent/failed 数据正确。
4. 前端：执行 `npm run type-check` 和 `npm run build`，验证新增编辑弹窗与图表参数类型正确。
5. 手动验证：创建/编辑 ApiKey、创建/编辑通知目标、在仪表盘切换整体/Topic/ApiKey 趋势。

## 7. 已确认产品口径

1. ApiKey 编辑范围限定为：名称、有效期、授权范围；完整 key 不支持查看，丢失时通过轮换生成新 key，旧 key 立即失效。
2. 通知目标编辑时类型只读：新增时可选类型，编辑时不能修改类型，只能改名称、地址、密钥、启用状态。
3. ApiKey 维度趋势只统计真实 ApiKey：仅统计带 `apiKeyId` 的调用；公开 Topic（NONE 模式）调用只进入整体和 Topic 维度，不进入 ApiKey 维度。

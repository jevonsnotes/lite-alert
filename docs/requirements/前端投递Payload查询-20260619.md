# 前端投递 Payload 查询需求分析

**日期**：2026-06-19  
**类型**：功能增强  
**关联模块**：可靠投递、投递记录、payload 权限、审计/调用记录、前端管理页面

## 1. 需求概述

当前后端已经将 Webhook 原始 payload 完整保存到 `la_notify_delivery.payload_json`，并提供了投递记录查询接口：

```text
GET /api/deliveries
GET /api/deliveries/{id}
```

但前端目前没有专门页面或入口展示投递记录和 payload。用户希望在前端页面中查询投递 payload，便于排查“某次 webhook 到底收到了什么内容、投递到了哪些目标、失败原因是什么”。

## 2. 技术可行性结论

结论：**可行，建议在现有“调用记录 / 审计日志”基础上增加投递详情入口，优先做轻量集成，不单独新建复杂模块。**

可行原因：

- 后端已经有 `NotifyDeliveryController` 查询投递记录。
- 后端已经有 payload 权限控制：管理员和拥有 `DELIVERY_PAYLOAD_READ` 的用户可查看完整 payload，普通用户返回脱敏 payload。
- 审计日志事件中可记录 `deliveryId`，可以从 `notify.sent / notify.failed / notify.give_up / notify.cancelled` 跳转或展开查看投递详情。
- 前端已有 `Audit.vue` 展开行展示 JSON 详情，可复用交互模式。

## 3. 推荐方案

### 3.1 MVP 方案：在调用记录页增加投递详情入口

推荐优先改造：`frontend/src/views/Audit.vue`

- 当审计事件中存在 `deliveryId` 时，展示“查看 Payload / 投递详情”按钮。
- 点击后调用：

```text
GET /api/deliveries/{deliveryId}
```

- 使用弹窗展示：
  - deliveryId
  - traceId
  - topicId
  - targetId
  - channel
  - status
  - attempt
  - nextRetryAt
  - lastError
  - payload
- payload 用 `<pre>` 或 JSON viewer 风格展示。
- 如果后端返回的是脱敏 payload，前端显示提示：

```text
当前账号无完整 Payload 查看权限，已展示脱敏内容。
```

### 3.2 后续增强：新增投递记录页面

后续可以新增路由：

```text
/deliveries
```

功能：

- 按状态筛选：PENDING / SENDING / RETRY_WAIT / SENT / GIVE_UP / CANCELLED
- 按 traceId 查询
- 按 Topic 查询
- 按通知目标查询
- 管理员手动重试 / 取消任务

本期不建议直接做复杂列表页，先从调用记录页的投递详情入口做起。

## 4. 风险点和注意事项

1. **payload 权限提示要明确**
   - 后端已经按权限返回完整或脱敏 payload。
   - 前端需要给用户明确提示，避免误以为原始 payload 本身就是星号。

2. **payload 内容可能很长**
   - 弹窗需要限制高度，内部滚动。
   - 避免撑破页面布局。

3. **payload 可能不是对象结构**
   - 后端可能返回字符串或脱敏文本。
   - 前端展示时要兼容非 JSON 内容。

4. **审计事件不一定都有 deliveryId**
   - `webhook.accepted` 可能只表示接收成功。
   - `notify.*` 事件更适合作为投递详情入口。

## 5. 预估工作量

| 范围 | 工作量 |
|------|--------|
| 调用记录页增加投递详情弹窗 | 0.5～1 天 |
| 新增完整投递记录页面 | 1～2 天 |

## 6. 推荐结论

建议先做 MVP：在 `Audit.vue` 中增加基于 `deliveryId` 的投递详情弹窗。确认体验后，再考虑新增独立投递记录页面。

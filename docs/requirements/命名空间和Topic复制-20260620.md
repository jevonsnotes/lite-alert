# 命名空间和 Topic 复制需求分析

**日期**：2026-06-20

## 需求概述

产品希望在命名空间列表和 Topic 列表中增加“复制”按钮，降低重复配置成本：

- 复制命名空间时，弹窗要求用户确认新命名空间名称，默认值为旧名称追加 `_copy`。
- 复制命名空间时，原命名空间下的 Topic 配置照搬到新命名空间下。
- 复制 Topic 时，同样弹窗要求用户确认新 Topic 名称，默认值为旧名称追加 `_copy`。
- 同时需要解释并修复“命名空间和 Topic 无法编辑名称”的问题。

## 技术可行性结论

可行。现有后端已经具备命名空间和 Topic 的创建、查询、保存能力，复制功能可以在 Service 层复用现有校验和持久化逻辑：

- 命名空间复制：新建一个命名空间，再遍历旧命名空间下的 Topic，逐个复制为新 Topic。
- Topic 复制：读取源 Topic 的配置，生成新 ID、新名称和 DRAFT 状态，保存到同一命名空间。
- 名称修改：在现有 PATCH 接口中扩展 `name` 字段，增加名称格式和唯一性校验。

## 推荐方案及理由

### 1. 后端接口设计

新增以下接口：

| 功能 | 接口 | 说明 |
| --- | --- | --- |
| 复制命名空间 | `POST /api/namespaces/{id}/copy` | 请求体包含新命名空间名称和描述，返回新命名空间 |
| 复制 Topic | `POST /api/topics/{id}/copy` | 请求体包含新 Topic 名称和描述，返回新 Topic |

扩展以下接口：

| 功能 | 接口 | 说明 |
| --- | --- | --- |
| 编辑命名空间名称/描述 | `PATCH /api/namespaces/{id}` | 支持 `name`、`description` |
| 编辑 Topic 名称/描述 | `PATCH /api/topics/{id}` | 支持 `name`、`description`，建议仅 DRAFT 可改名 |

理由：

- 复制是明确业务动作，用独立 `/copy` 接口比前端拼多个创建请求更可靠。
- 名称修改继续走 PATCH，符合已有接口风格。
- 复制逻辑放在 Service 层，可以统一处理权限、唯一性、审计日志和默认状态。

### 2. 复制规则

#### 命名空间复制

- 新命名空间：
  - `id` 重新生成。
  - `name` 使用用户弹窗确认的名称，默认 `旧名称_copy`。
  - `description` 默认复用旧描述。
  - `ownerId` 使用当前用户或源命名空间归属策略，推荐使用当前用户。
  - `status` 默认为 `ACTIVE`。
- Topic：
  - 复制源命名空间下全部 Topic 的配置。
  - 每个 Topic 重新生成 `id`。
  - `namespaceId` 指向新命名空间。
  - `namespaceName` 使用新命名空间名称。
  - `name` 保持不变。
  - `description`、`auth`、`inboundFormat`、`templates` 等配置照搬。
  - `status` 由复制弹窗选择：
    - 复制为草稿：新 Topic 统一置为 `DRAFT`。
    - 保持原状态：新 Topic 沿用源 Topic 的 `DRAFT` / `PUBLISHED` / `DISABLED` 状态。
  - 订阅关系按产品确认需要复制：源 Topic 已订阅的通知目标同步复制到新 Topic。

#### Topic 复制

- 新 Topic：
  - `id` 重新生成。
  - `namespaceId` 与源 Topic 相同。
  - `name` 使用用户弹窗确认的名称，默认 `旧名称_copy`。
  - `description`、`auth`、`inboundFormat`、`templates` 等配置照搬。
  - `status` 由复制弹窗选择：复制为 `DRAFT` 或保持源 Topic 状态。
  - 订阅关系按产品确认需要复制：源 Topic 已订阅的通知目标同步复制到新 Topic。

### 3. 前端交互设计

- `frontend/src/views/Namespaces.vue`
  - 命名空间操作列新增“复制”。
  - Topic 内嵌表格操作列新增“复制”。
  - 命名空间复制弹窗：名称默认 `旧名称_copy`，描述默认旧描述，并提供“复制为草稿 / 保持原状态”的 Topic 状态选项。
  - Topic 复制弹窗：名称默认 `旧名称_copy`，描述默认旧描述，并提供“复制为草稿 / 保持原状态”的状态选项。
  - 复制操作会同时复制订阅关系。
  - 成功后刷新列表，并提示“已复制”。
- `frontend/src/views/TopicDetail.vue`
  - 基础信息区补充 Topic 名称编辑能力。
  - DRAFT 状态允许修改名称；PUBLISHED/DISABLED 状态展示为只读或禁用，并提示“名称影响 Webhook URL，发布后不可修改”。

## 风险点和注意事项

1. **Webhook URL 变化风险**
   - 命名空间名和 Topic 名都会影响 `/api/webhook/{namespace}/{topic}`。
   - 已发布 Topic 改名可能导致调用方 URL 失效，因此推荐限制 Topic 仅 DRAFT 可改名。

2. **命名空间改名需要同步 Topic 冗余字段**
   - Topic 表中保存了 `namespaceName`，Webhook 查询会使用该字段。
   - 命名空间改名后必须同步更新该命名空间下所有 Topic 的 `namespaceName`。

3. **名称唯一性**
   - 命名空间名称全局唯一。
   - Topic 名称在同一命名空间内唯一。
   - 默认 `_copy` 名称可能已存在，后端必须返回明确错误，前端提示用户换名。

4. **复制后的状态**
   - 产品确认复制时需要允许用户选择“复制为草稿”或“保持原状态”。
   - 如果选择保持原状态，新复制出的 PUBLISHED Topic 会立即拥有可调用的新 Webhook URL，需要在弹窗中明确提示风险。

5. **复制订阅关系风险**
   - 产品确认复制时需要复制订阅关系。
   - 如果新 Topic 保持 PUBLISHED 状态，复制后可能立即按同一批通知目标投递消息；前端需要提示用户确认。

## 预估工作量

- 后端复制与名称编辑接口：0.5–1 天。
- 后端测试用例：0.5 天。
- 前端复制弹窗、按钮和名称编辑：0.5–1 天。
- 联调和验证：0.5 天。

整体预估：约 1.5–3 个工作日，取决于是否要求复制订阅关系、ApiKey 覆盖关系等外围关联数据。

## 产品确认结果

1. 复制后的 Topic 状态：复制弹窗允许用户选择“复制为草稿”或“保持原状态”。
2. 复制命名空间 / Topic 时：需要复制 Topic 的订阅关系。
3. 名称编辑规则：采用受控改名。
   - 命名空间允许改名，并提示会影响该空间下所有 Topic 的 Webhook URL。
   - Topic 仅 DRAFT 状态允许改名，PUBLISHED / DISABLED 状态不可改名。

# 接口鉴权与角色权限模型改造需求分析

**日期**：2026-06-19  
**类型**：安全模型 / 权限体系 / 前端管理能力改造  
**关联模块**：Webhook 接入、Topic 鉴权配置、ApiKey、用户管理、角色管理、权限授权、前端路由与菜单、数据库迁移

## 1. 需求概述

本次需求包含三部分：

1. **Webhook 接口去掉免认证模式**
   - 当前 Topic 鉴权支持 `API_KEY` 和 `NONE`。
   - 新需求要求移除 `NONE` 免认证模式，所有 Webhook 调用都必须携带认证信息。
   - 认证方式保留 ApiKey 体系，但支持两种传递方式：
     - 请求头认证：例如 `Authorization: Bearer xxxxx`。
     - URL 参数认证：例如 `/api/webhook/abc/topic?key=xxxxx`。
   - 普通用户也可以在 Topic 配置中自由切换“请求头认证 / 参数认证”。

2. **授权改为树状结构**
   - 当前用户 payload 查看权限是扁平权限字段。
   - 新需求要求授权改为树状结构，通过勾选给用户或角色授权。
   - 权限树应能覆盖菜单权限、功能权限、数据能力权限，例如：
     - 系统管理
       - 用户管理
       - 角色管理
     - 投递记录
       - 查看投递列表
       - 查看完整 Payload
     - Topic 管理
       - 创建 Topic
       - 发布 Topic

3. **增加角色配置菜单**
   - 管理员可以维护角色。
   - 权限与角色绑定。
   - 用户与角色绑定。
   - 后续用户拥有的权限由其角色集合汇总得出，不再直接只靠 `ADMIN / USER` 两态和用户自身 permissions 字段。

通俗理解：

> 现在是“用户身上直接贴权限标签”；改造后是“先定义角色，给角色配置权限，再把用户放进角色里”。这样权限更清晰，也更适合多人协作和后续扩展。

## 2. 当前实现现状

### 2.1 Topic 鉴权模式

当前 `Topic` 中定义：

```java
public enum AuthMode { API_KEY, NONE }
```

当前 Webhook 服务中：

- `API_KEY` 模式才调用 `ApiKeyAuthenticator.authenticate(...)`。
- `NONE` 模式不要求 Authorization，只依赖 IP 白名单、限流、大小限制、Schema 校验。
- 公开 Topic 的大小限制走 `publicMaxBodySize`。

涉及文件：

- `backend/src/main/java/io/litealert/topic/domain/Topic.java`
- `backend/src/main/java/io/litealert/webhook/WebhookService.java`
- `backend/src/main/java/io/litealert/webhook/ApiKeyAuthenticator.java`
- `frontend/src/views/TopicDetail.vue`
- `docs/design/07-webhook-api.md`
- `docs/design/11-apikey.md`

### 2.2 当前用户权限模型

当前用户实体已有：

```java
public enum Role { ADMIN, USER }
public enum Permission { DELIVERY_PAYLOAD_READ }
```

即当前仍是：

- 用户有一个简单角色：`ADMIN / USER`。
- 用户可以额外直接挂权限：`permissions_json`。

这能满足少量权限，但不适合长期维护复杂权限矩阵。

涉及文件：

- `backend/src/main/java/io/litealert/auth/domain/User.java`
- `backend/src/main/java/io/litealert/auth/domain/UserStore.java`
- `backend/src/main/java/io/litealert/auth/web/UserController.java`
- `frontend/src/views/AdminUsers.vue`

### 2.3 当前前端菜单

当前管理员菜单由 `auth.isAdmin` 控制：

- 用户管理
- 系统设置

涉及文件：

- `frontend/src/layouts/AppLayout.vue`
- `frontend/src/router/index.ts`
- `frontend/src/stores/auth.ts`

## 3. 技术可行性结论

结论：**可行，但属于中等规模安全模型改造，建议分阶段实施。**

可行原因：

- 项目已经数据库化，并引入 Flyway，适合新增角色表、角色权限表、用户角色表。
- 当前权限判断集中在 `CurrentUser`、Controller 和部分 Service 中，可以逐步抽象为统一权限判断。
- ApiKey 鉴权已经独立成模块，只需要扩展“从哪里取 key”的逻辑，不需要重写 ApiKey 本身。
- 前端已有 Element Plus，支持 Tree、Checkbox、Dialog、Table，适合实现权限树配置。

主要难点：

1. **兼容现有 Topic 的 NONE 模式**
   - 旧数据里可能已有 `auth.mode=NONE`。
   - 新版本需要迁移为认证模式，或者启动后将 NONE 视为默认 HEADER/API_KEY。

2. **权限模型切换不能把管理员锁死**
   - Flyway 初始化时必须创建内置管理员角色。
   - 默认 admin 用户必须绑定管理员角色。
   - 管理员角色必须拥有全部权限。

3. **前端路由、菜单、接口权限要统一**
   - 不能只隐藏菜单，后端接口也必须校验权限。
   - 前端菜单权限只做体验，后端才是安全边界。

4. **URL 参数传 key 有泄露风险**
   - URL 可能出现在浏览器历史、代理日志、网关日志中。
   - 但用户明确要求支持，所以需要在 UI 和文档中提示风险。

## 4. 推荐方案

### 4.1 Webhook 鉴权模式改造

建议将 Topic 的认证配置从：

```java
AuthMode { API_KEY, NONE }
```

改为：

```java
AuthMode { API_KEY }
AuthCredentialTransport { HEADER, QUERY }
```

或者更直接：

```java
public enum AuthMode { API_KEY }
public enum KeyLocation { HEADER, QUERY }
```

Topic.Auth 推荐字段：

| 字段 | 含义 |
|------|------|
| `mode` | 固定为 `API_KEY`，保留字段方便未来扩展 |
| `keyLocation` | `HEADER` 或 `QUERY` |
| `queryParamName` | 参数认证时的参数名，默认 `key` |
| `ipWhitelist` | IP 白名单 |
| `rateLimit` | 限流配置 |

#### 请求头认证

```http
POST /api/webhook/{namespace}/{topic}
Authorization: Bearer <apiKey>
```

#### 参数认证

```http
POST /api/webhook/{namespace}/{topic}?key=<apiKey>
```

默认建议：

- 新建 Topic 默认使用 `HEADER`。
- 用户可切换为 `QUERY`。
- 参数名默认 `key`，后续可配置；第一版也可以固定为 `key`。

### 4.2 去掉 NONE 模式

后端：

- `WebhookService` 不再允许绕过 ApiKey 鉴权。
- 所有 Topic 都必须执行 ApiKey 校验。
- 删除或废弃 `publicMaxBodySize` 和 `allowUserPublicTopic` 相关逻辑。
- audit 不再记录 `authMode=NONE`。

前端：

- Topic 配置页去掉“免认证 / NONE”选项。
- 改为认证位置选择：

```text
ApiKey 传递方式：
[请求头 Authorization] [URL 参数 key]
```

文档生成：

- HEADER 模式生成：

```bash
curl -X POST '.../api/webhook/ns/topic' \
  -H 'Authorization: Bearer <apiKey>'
```

- QUERY 模式生成：

```bash
curl -X POST '.../api/webhook/ns/topic?key=<apiKey>'
```

### 4.4 角色模型设计

新增表：`la_role`

| 字段 | 含义 |
|------|------|
| `id` | 角色 ID |
| `name` | 角色名称 |
| `description` | 描述 |
| `system_builtin` | 是否系统内置 |
| `permissions_json` | 权限编码列表 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

新增表：`la_user_role`

| 字段 | 含义 |
|------|------|
| `user_id` | 用户 ID |
| `role_id` | 角色 ID |

内置角色：

1. `SUPER_ADMIN`
   - 拥有全部权限。
   - 默认 admin 用户绑定。
   - 不允许删除。

2. `NORMAL_USER`
   - 普通用户默认角色。
   - 可由管理员调整权限，也可设为系统内置不允许删除。

是否保留 `User.Role ADMIN/USER`：

- 第一阶段建议保留，作为兼容字段和粗粒度标识。
- 真正权限判断逐步改为角色权限。
- 后续稳定后可再废弃 `User.Role`。

### 4.5 后端权限判断

新增服务：`PermissionService`

职责：

- 根据当前用户 ID 查询其角色列表。
- 汇总角色权限。
- 提供判断方法：

```java
boolean has(String permission)
void require(String permission)
```

`CurrentUser` 可增加：

```java
Set<String> permissions()
boolean hasPermission(String permission)
```

后端 Controller / Service 中逐步替换：

```java
currentUser.isAdmin()
```

为：

```java
permissionService.require("ROLE_UPDATE")
```

注意：

- 第一阶段可以先在新增角色管理、payload 查询、系统设置等关键接口启用权限校验。
- 旧接口逐步迁移，避免一次性改动过大。

### 4.6 前端改造

新增菜单：

```text
角色管理
```

建议路径：

```text
/admin/roles
```

角色管理页面能力：

- 角色列表
- 新建角色
- 编辑角色名称 / 描述
- 权限树勾选
- 删除非内置角色

用户管理页面改造：

- 去掉或弱化直接权限勾选。
- 改为给用户绑定角色。
- 可多选角色。

菜单显示：

- 当前登录用户返回权限列表。
- 前端根据权限决定菜单是否显示。
- 后端接口仍做最终权限校验。

## 5. 数据迁移方案

Flyway 新增 V3 或后续版本：

1. 新增 `la_role`。
2. 新增 `la_user_role`。
3. 插入内置角色：
   - `r_super_admin`
   - `r_normal_user`
4. 给默认 `u_admin` 绑定 `r_super_admin`。
5. 给已有 ADMIN 用户绑定 `r_super_admin`。
6. 给已有 USER 用户绑定 `r_normal_user`。
7. 旧 `permissions_json` 可以迁移到用户专属角色，或暂时保留兼容。
8. 将旧 Topic 中 `auth.mode=NONE` 的配置迁移为 `API_KEY + HEADER` 或 `API_KEY + QUERY`。

关于 NONE 迁移建议：

- 安全优先：统一迁移为 `API_KEY + HEADER`。
- 用户需要手动给调用方分配 ApiKey。
- 前端在 Topic 列表或详情中提示“原免认证 Topic 已升级为 ApiKey 认证”。

## 6. 风险点和注意事项

1. **移除 NONE 会影响现有公开 Webhook 调用方**
   - 原本不带 key 的调用会开始失败。
   - 需要发版说明和迁移提示。

2. **URL 参数认证有泄露风险**
   - URL 参数可能被代理、网关、浏览器历史记录保存。
   - UI 需要提示“更推荐请求头认证”。

3. **角色权限容易误配导致管理员失权**
   - 内置超级管理员角色不能删除。
   - 至少保留一个拥有角色管理权限的用户。

4. **权限改造范围大**
   - 如果一次性替换所有 `isAdmin()`，风险较高。
   - 建议分阶段：先建模型和页面，再逐步接入关键接口。

5. **前端菜单隐藏不是安全边界**
   - 后端必须校验权限。

## 7. 推荐实施阶段

### 阶段一：Webhook 鉴权模式改造

- 删除 NONE 选项。
- Topic Auth 增加 `keyLocation=HEADER|QUERY`。
- Webhook 支持从 Header 或 query 参数取 ApiKey。
- 普通用户可切换认证传递方式。
- 更新接入文档生成。
- 增加兼容旧 Topic NONE 的迁移逻辑。

### 阶段二：角色和权限模型

- 新增角色表和用户角色表。
- Flyway 初始化内置角色。
- 当前用户信息返回 permissions。
- 新增 `PermissionService`。
- payload 查询改为角色权限判断。

### 阶段三：前端角色管理和用户绑定角色

- 新增角色配置菜单。
- 角色页面支持权限树勾选。
- 用户管理页支持绑定角色。
- 前端菜单按权限显示。

### 阶段四：逐步替换后端权限判断

- 将用户管理、角色管理、系统设置、投递 payload、ApiKey、Topic 等接口逐步接入权限码。
- 补充测试覆盖权限允许 / 拒绝场景。

## 8. 预估工作量

| 阶段 | 内容 | 粗略工作量 |
|------|------|------------|
| 阶段一 | 去 NONE、Header/Query ApiKey、Topic 页面和文档生成 | 1.5～2.5 天 |
| 阶段二 | 角色/权限表、PermissionService、Flyway 迁移 | 1.5～2.5 天 |
| 阶段三 | 角色管理页面、用户绑定角色、权限树 UI | 2～3 天 |
| 阶段四 | 后端接口权限逐步替换和测试 | 2～4 天 |

整体建议按阶段实施，避免一次性改动导致鉴权链路不可控。

## 9. 推荐结论

推荐确认以下产品口径后进入开发：

1. Webhook 不再支持免认证。
2. 所有 Topic 默认使用 ApiKey 请求头认证。
3. 用户可切换为 URL 参数认证，默认参数名为 `key`。
4. URL 参数认证在 UI 中提示风险。
5. 角色权限模型采用：权限树 → 角色 → 用户。
6. 默认 admin 绑定内置超级管理员角色。
7. 旧 NONE Topic 统一迁移为 `API_KEY + HEADER`。

确认后建议先实施阶段一和阶段二，再做前端角色管理页面。

# 数据库存储与命名空间状态改造需求分析

**日期**：2026-06-18
**类型**：架构升级 + 功能增强
**关联模块**：命名空间、Topic、Webhook、审计日志、认证、用户管理、存储层、部署配置

## 1. 需求概述

本次需求包含四部分：

1. **命名空间增加状态管理**
   - 命名空间列表增加状态列。
   - 操作列增加“禁用 / 恢复”按钮。
   - 禁用命名空间后，该命名空间下所有 Topic 都无法通过 Webhook 访问。
   - 禁用命名空间时不修改 Topic 自身状态，Topic 仍保持原来的 `DRAFT / PUBLISHED / DISABLED`。

2. **审计日志增强**
   - 命名空间禁用、恢复事件需要记录审计日志。
   - Topic 禁用、恢复事件需要记录审计日志。

3. **存储层从文件改造为数据库**
   - 当前文件存储存在一致性、并发和维护风险，需要改为数据库存储。
   - 默认数据库使用 H2。
   - 需要支持 PostgreSQL、MySQL、GaussDB、OceanBase。
   - 使用 MyBatis-Flex 框架。
   - 系统启动时，如果数据库未初始化，需要自动初始化所有表结构和基础数据。

4. **用户密码不可逆存储**
   - 后端只保存加密后的密码，不保存明文或可逆密文。
   - 前端先对密码做 MD5，再发送到后端。
   - 后端收到 MD5 后，再使用 Spring Security 密码套件加密入库。
   - 登录校验时，前端同样先 MD5，后端使用 Spring Security 密码套件校验该 MD5 值。

## 2. 与现有项目规范的冲突

当前项目规范与设计文档中存在以下约束：

- `CLAUDE.md`：项目目标是“无数据库化、文件加密存储”。
- `docs/design/02-data-model.md`：数据模型围绕 `/data/*.json` 文件设计。
- `README.md`：强调“无外部数据库 / 消息队列依赖”。

本次需求明确要求改为数据库，因此需要同步更新：

- `CLAUDE.md`
- `README.md`
- `docs/design/02-data-model.md`
- `docs/design/09-deploy.md`
- `docker/docker-compose.yml`
- `application.yml / application-prod.yml`

建议确认后，将项目定位从：

> 无数据库化、文件加密存储

调整为：

> 默认内置 H2、可切换外部数据库的轻量级消息通知服务

这样既保留“开箱即用”的轻量体验，又能支持生产数据库。

## 3. 技术可行性结论

结论：**可行，但属于中高风险架构改造，建议分阶段实施。**

可行原因：

- 后端已有清晰的 Store 层，例如 `UserStore / NamespaceStore / TopicStore / ApiKeyStore / NotifyTargetStore / SubscriptionStore`。
- 大部分业务代码通过 Store 访问数据，理论上可以将 Store 实现从文件替换为数据库。
- Spring Boot 对 H2 / PostgreSQL / MySQL 等数据源支持成熟。
- MyBatis-Flex 可以作为数据库访问框架，替代当前文件 Store 的读写逻辑。

主要难点：

- 当前实体中存在复杂 JSON 字段，例如：
  - Topic inboundFormat
  - Topic auth
  - Topic templates
  - ApiKey scopes
  - Subscription contactIds
- 审计日志当前是 JSON-lines 文件，如果也迁入数据库，需要新增审计表和查询逻辑。
- 需要兼容多数据库方言，H2、PostgreSQL、MySQL、GaussDB、OceanDB/OceanBase 的 DDL 细节不同。
- 现有文件数据是否迁移到数据库，需要明确策略。

## 4. 推荐总体方案

### 4.1 数据库支持策略

推荐使用：

- **MyBatis-Flex**：负责 ORM / Mapper / 查询。
- **HikariCP**：Spring Boot 默认连接池。
- **数据库初始化器**：应用启动时检查表是否存在，不存在则执行对应数据库方言的初始化脚本。

建议支持的数据库模式：

| 数据库 | 推荐模式 | 说明 |
| --- | --- | --- |
| H2 | 默认嵌入式 | 开箱即用，适合本地和轻量部署 |
| PostgreSQL | 原生 PostgreSQL 方言 | 生产推荐之一 |
| MySQL | MySQL 8+ 方言 | 生产推荐之一 |
| GaussDB | PostgreSQL 兼容模式优先 | 具体取决于部署的 GaussDB 版本 |
| OceanBase | MySQL 兼容模式 | 本期按 OceanBase MySQL 兼容模式适配 |

确认口径：OceanBase 按 MySQL 兼容模式支持；GaussDB 按 PostgreSQL 兼容模式支持。

### 4.2 初始化方式

推荐目录：

```text
backend/src/main/resources/db/schema-h2.sql
backend/src/main/resources/db/schema-postgresql.sql
backend/src/main/resources/db/schema-mysql.sql
backend/src/main/resources/db/schema-gaussdb.sql
backend/src/main/resources/db/schema-oceanbase.sql
backend/src/main/resources/db/data-default.sql
```

启动流程：

1. 读取配置中的数据库类型，例如：
   - `lite-alert.database.type=h2|postgresql|mysql|gaussdb|oceanbase`
2. 连接数据库。
3. 检查核心表是否存在，例如 `la_schema_version`。
4. 如果不存在：
   - 执行对应数据库的 schema 脚本。
   - 初始化默认系统配置。
   - 初始化默认管理员。
5. 如果存在：
   - 检查 schema version。
   - 后续可按版本执行迁移脚本。

### 4.3 表结构建议

核心表建议：

| 表名 | 说明 |
| --- | --- |
| `la_schema_version` | 数据库初始化和迁移版本 |
| `la_user` | 用户 |
| `la_namespace` | 命名空间 |
| `la_topic` | Topic |
| `la_api_key` | ApiKey |
| `la_notify_target` | 通知目标 |
| `la_subscription` | Topic 与通知目标订阅关系 |
| `la_audit_log` | 审计日志 |
| `la_system_settings` | 系统设置 |

复杂结构字段建议先用 JSON 文本保存，降低迁移复杂度：

- `la_topic.auth_json`
- `la_topic.inbound_format_json`
- `la_topic.templates_json`
- `la_api_key.scopes_json`
- `la_subscription.contact_ids_json`

理由：

- 当前系统本身就以 JSON 对象表达这些字段。
- 多数据库都可以保存 JSON 字符串。
- 后续如果某些字段需要高频查询，再拆成关系表。

## 5. 命名空间状态设计

### 5.1 数据模型

`Namespace` 增加：

```java
public enum Status { ACTIVE, DISABLED }
private Status status;
private Instant disabledAt;
private String disabledBy;
```

数据库字段：

```sql
status varchar(16) not null default 'ACTIVE'
disabled_at timestamp null
disabled_by varchar(64) null
```

### 5.2 后端行为

新增接口：

```text
POST /api/namespaces/{id}/disable
POST /api/namespaces/{id}/enable
```

禁用逻辑：

- 只修改 Namespace 状态为 `DISABLED`。
- 不修改其下 Topic 状态。
- 写审计日志：`namespace.disable`。

恢复逻辑：

- 修改 Namespace 状态为 `ACTIVE`。
- 写审计日志：`namespace.enable`。

Webhook 访问逻辑：

- 在 `WebhookService` 中找到 Topic 后，需要检查该 Topic 所属 Namespace 是否为 `DISABLED`。
- 如果命名空间禁用，则拒绝访问。
- 建议新增错误码：`NAMESPACE_DISABLED`，HTTP 状态建议 `423 Locked`。

### 5.3 前端行为

命名空间列表新增：

- 状态列：`ACTIVE / DISABLED`
- 操作列：
  - ACTIVE：显示“禁用”
  - DISABLED：显示“恢复”

禁用提示：

> 禁用命名空间后，该空间下所有 Topic 的 Webhook 调用都会被拒绝，但 Topic 状态不会改变。确定继续？

## 6. 审计日志增强

当前 Topic 禁用已有审计：

- `topic.disable`

但 Topic 恢复当前缺少审计，需要补充：

- `topic.enable`

命名空间新增：

- `namespace.disable`
- `namespace.enable`

建议审计字段：

```json
{
  "actor": "u_xxx",
  "namespaceId": "ns_xxx",
  "topicId": "t_xxx",
  "name": "xxx"
}
```

数据库化后建议审计日志也进入 `la_audit_log`，字段：

| 字段 | 说明 |
| --- | --- |
| id | 主键 |
| ts | 时间 |
| type | 事件类型 |
| actor | 操作人 |
| trace_id | TraceId |
| attrs_json | 事件扩展字段 JSON |

## 7. 密码方案分析

### 7.1 当前状态

当前后端已经使用 Spring Security `PasswordEncoder` 进行密码哈希：

- 登录：`passwordEncoder.matches(rawPassword, passwordHash)`
- 创建用户：`passwordEncoder.encode(req.password())`
- 重置密码：`passwordEncoder.encode(req.password())`

但 `User.passwordHash` 字段当前带有 `@Encrypted`，在文件存储时会再套一层 Jasypt 加密。

数据库化后建议：

- 移除密码字段的可逆加密语义。
- 数据库只保存 Spring Security 生成的不可逆哈希。

### 7.2 前端 MD5 + 后端 Spring Encoder

按你的要求，最终链路为：

```text
用户输入原始密码
  -> 前端 MD5(password)
  -> 后端接收 passwordMd5
  -> 后端 PasswordEncoder.encode(passwordMd5)
  -> 数据库保存哈希
```

登录校验：

```text
用户输入原始密码
  -> 前端 MD5(password)
  -> 后端 PasswordEncoder.matches(passwordMd5, storedHash)
```

注意事项：

- MD5 不是密码安全算法，但这里只作为“前端传输前摘要”，最终安全性仍依赖后端 BCrypt/DelegatingPasswordEncoder。
- HTTPS 仍然必须使用，不能因为前端 MD5 就认为传输安全。
- 因为后端存的是 `MD5(password)` 的 BCrypt 结果，所以老的明文密码 BCrypt 存量用户需要迁移或重置密码。

### 7.3 存量密码迁移策略

可选方案：

1. **强制重置密码**（推荐）
   - 数据库初始化后，默认管理员按新规则写入。
   - 从旧文件迁移用户时不迁移旧密码，要求管理员重置。

2. **兼容一次登录后升级**
   - 后端同时识别旧 hash 与新 hash。
   - 用户首次成功登录后，将新 MD5 链路 hash 写回。
   - 实现更复杂。

推荐本期选择方案 1，降低安全和实现风险。

## 8. 文件数据迁移策略

本次数据库改造涉及已有 `/data/*.json` 文件。建议选择以下策略之一：

### 方案 A：仅初始化空数据库，不迁移旧文件（简单）

- 新版本启动时，如果数据库为空，创建表和默认管理员。
- 旧 JSON 文件不自动导入。
- 用户需要重新配置 Namespace、Topic、ApiKey、通知目标。

优点：实现快、风险低。  
缺点：已有数据不能保留。

### 方案 B：自动迁移旧文件到数据库（推荐，但工作量更大）

- 启动时检测数据库为空，同时检测旧 `/data/*.json`。
- 将用户、命名空间、Topic、ApiKey、通知目标、订阅、系统设置迁移到数据库。
- 审计日志可选择不迁移，或仅迁移近期日志。
- 迁移完成后写入 `la_schema_version` 和迁移标记，避免重复导入。

优点：用户无感升级。  
缺点：需要大量测试，尤其是加密字段、JSON 字段和 ApiKey 哈希。

确认口径：**不迁移旧 `/data/*.json` 数据**。新版本以空数据库初始化为准，自动创建表结构和默认管理员；旧文件仅作为历史备份，不自动导入。

## 9. 推荐分阶段实施

### 阶段 1：命名空间状态与审计增强

- Namespace 增加状态。
- 增加禁用/恢复接口。
- Webhook 检查 Namespace 状态。
- Topic enable 增加审计。
- 前端命名空间列表增加状态、禁用/恢复按钮。
- 继续使用当前文件 Store。

收益：较快满足业务控制诉求。

### 阶段 2：引入数据库基础设施

- 引入 MyBatis-Flex、H2、PostgreSQL、MySQL 驱动。
- 增加数据库配置。
- 增加数据库初始化器。
- 建表脚本和版本表。
- 新增数据库 Mapper / Entity 基础结构。

### 阶段 3：Store 层逐步替换

建议顺序：

1. UserStore
2. NamespaceStore
3. TopicStore
4. ApiKeyStore
5. NotifyTargetStore
6. SubscriptionStore
7. SystemSettingsService
8. AuditLogger / AuditController / StatsController

### 阶段 4：密码链路改造

- 前端引入 MD5 工具。
- 登录、新建用户、重置密码统一提交 MD5。
- 后端字段语义改为 `passwordHash = PasswordEncoder(MD5(password))`。
- 文档明确“前端 MD5 不是替代 HTTPS”。

### 阶段 5：数据迁移与兼容测试

- 如果选择自动迁移旧文件，增加迁移器。
- 增加 H2 集成测试。
- 至少增加 MySQL/PostgreSQL 方言初始化测试。

## 10. 风险点和注意事项

1. **多数据库方言风险**
   - H2、MySQL、PostgreSQL、GaussDB、OceanBase 的 DDL 和 JSON 类型能力不同。
   - 推荐核心 JSON 字段先用 `TEXT/CLOB`，不要依赖数据库 JSON 类型。

2. **密码迁移风险**
   - 当前用户密码 hash 与新 MD5 链路不兼容。
   - 必须确认是否强制重置密码或做兼容登录迁移。

3. **审计统计改造风险**
   - 当前仪表盘基于文件 audit log 扫描。
   - 审计迁入数据库后，StatsController 需要改成 SQL 聚合，性能更好但改动较多。

4. **一次性全量改造风险高**
   - 如果同时改数据库、认证、命名空间状态、审计、迁移，回归范围非常大。
   - 建议按阶段推进。

## 11. 预估工作量

粗略估计：

| 阶段 | 工作量 |
| --- | --- |
| 命名空间状态 + 审计增强 | 0.5 - 1 天 |
| 数据库基础设施 + H2 初始化 | 1 - 2 天 |
| Store 层替换为 MyBatis-Flex | 3 - 5 天 |
| 多数据库方言脚本适配 | 1 - 2 天 |
| 密码 MD5 + Spring Encoder 改造 | 0.5 - 1 天 |
| 旧文件数据自动迁移 | 不实施 |
| 测试与回归 | 2 - 3 天 |

总计：**不迁移旧数据约 7 - 10 天；自动迁移旧数据约 10 - 15 天。**

## 12. 已确认口径

1. `oceandb` 更正为 **OceanBase**，本期按 **MySQL 兼容模式** 支持。
2. GaussDB 按 **PostgreSQL 兼容模式** 支持。
3. 旧 `/data/*.json` 数据 **不自动迁移** 到数据库。
4. 存量用户密码采用 **前置重置**，不做旧密码兼容登录。
5. 采用分阶段实施，但全部阶段都要完成；后续不再逐阶段询问是否继续。

# 消息可靠投递持久化队列需求分析

**日期**：2026-06-19  
**类型**：可靠性增强 / 架构优化  
**关联模块**：Webhook 接入、通知派发、数据库持久化队列、Flyway 数据库迁移、系统设置、用户权限、审计日志

## 1. 需求概述

本需求明确采用 **数据库实现可靠投递队列**，不引入 Kafka / RabbitMQ / Redis 等外部消息队列。

当前 Lite-Alert 的 Webhook 接收流程是：请求校验通过后写入 `webhook.accepted` 审计日志，然后把通知派发交给内存异步线程池执行。通知失败后，会进入内存 `DelayQueue` 做指数退避重试。

这种模式响应快，但存在生产风险：

> 如果系统在“已接收但尚未发送”或“等待重试”期间宕机，内存中的待投递任务会丢失。数据库中可能只能看到 `webhook.accepted` 或 `notify.failed`，但系统重启后不会自动补发。

本次需求目标是新增 **数据库持久化投递队列 + Flyway 数据库升级能力 + 投递数据权限化查询**，让系统做到：

1. Webhook 消息校验通过后，把每个通知目标的投递任务保存到数据库。
2. Webhook 成功接收后返回 **HTTP 200**，避免部分调用方把 202 识别为失败。
3. 后台 Worker 从数据库领取待发送任务并投递。
4. 发送失败后，更新数据库中的重试次数和下次重试时间。
5. 系统重启后，可以继续扫描数据库中未完成的任务并恢复投递。
6. 支持多实例安全领取任务，避免同一任务被多个实例同时处理。
7. 支持投递任务保留周期在系统配置页配置，修改后立即生效。
8. 完整记录 payload；管理员可授权用户查看完整 payload；无权限用户只能查看脱敏 payload。
9. 引入 Flyway 支持数据库 schema 自动升级：新版本部署后自动执行缺失的升级脚本。
10. 审计日志继续记录接收、发送成功、发送失败、最终放弃等事件，便于前端追踪。

通俗理解：

- 现在像“把任务写在白板上，服务一重启白板就没了”；
- 新方案是“把任务写进数据库待办表，服务重启后还能继续处理”；
- 同时用 Flyway 给数据库加“自动升级机制”，以后新增表、字段、索引时，新版本应用能自己补齐。

## 2. 用户确认的关键约束

本版方案按以下产品约束调整：

1. **成功接收返回 200，不返回 202**
   - 原因：部分外部系统只把 200 识别为成功，收到 202 可能会误判失败并重复调用或告警。
   - 调整后：只要 Webhook 校验通过并且投递任务落库成功，就返回 `200 OK`。

2. **不处理“外部已发送成功但未更新 SENT 就宕机”的极端重复场景**
   - 本方案仍保留 `deliveryId / traceId` 便于追踪和下游幂等，但不把这个极端场景作为本期重点风险闭环。
   - 产品口径聚焦为：解决“已接收但未发送 / 待重试任务因系统重启丢失”的问题。

3. **投递任务保留周期必须可配置且立即生效**
   - 系统设置页新增“投递记录保留”配置。
   - 修改后不需要重启，后台清理任务下次执行时读取最新配置。

4. **完整 payload 要落库并支持权限化查询**
   - 数据库中完整保存 `payload_json`。
   - 管理员可以给用户授权“查看完整投递 payload”。
   - 有权限用户返回完整 payload。
   - 无权限用户返回脱敏 payload。

5. **数据库升级引入 Flyway**
   - 不再自研迁移器。
   - 使用 Flyway 管理版本化 DDL。
   - 现有 `DatabaseInitializer` 后续只负责基础数据 bootstrap，不再负责表结构版本升级。

## 3. 当前实现现状

### 3.1 接收链路

- Webhook 入口：`backend/src/main/java/io/litealert/webhook/web/WebhookController.java`
- 核心处理：`backend/src/main/java/io/litealert/webhook/WebhookService.java`

当前 `WebhookService.handle()` 在校验成功后执行：

```java
audit.log("webhook.accepted", ctxAttrs);
dispatcher.dispatchAsync(nctx);
```

Controller 当前返回：

```java
ResponseEntity.accepted().body(ack)
```

也就是 HTTP `202 Accepted`。

新方案需要改为：

```java
ResponseEntity.ok(body)
```

也就是 HTTP `200 OK`。

### 3.2 派发链路

- 派发类：`backend/src/main/java/io/litealert/notify/NotifyDispatcher.java`
- 异步线程池：`backend/src/main/java/io/litealert/notify/AsyncConfig.java`

当前使用：

```java
@Async("notifyExecutor")
public void dispatchAsync(NotifyContext ctx)
```

线程池配置为：

- core：4
- max：16
- queue：1000

### 3.3 重试链路

- 重试类：`backend/src/main/java/io/litealert/notify/RetryQueue.java`

当前重试队列是：

```java
private final DelayQueue<DelayedTask> queue = new DelayQueue<>();
```

属于 JVM 内存队列，重启后不会保留。

重试节奏：

```text
1m → 5m → 30m → 2h → 6h
```

最多 5 次后写 `notify.give_up`。

### 3.4 当前数据库初始化 / 升级能力

当前数据库初始化类：

- `backend/src/main/java/io/litealert/common/db/DatabaseInitializer.java`

当前能力：

1. 启动时检查 `la_schema_version` 是否存在且有记录。
2. 如果不存在，认为是全新数据库，执行对应数据库类型的完整 schema 脚本。
3. 初始化完成后插入 `la_schema_version(version=1)`。

当前限制：

> 现在只有“新库初始化能力”，没有“老库升级能力”。

新方案引入 Flyway 后：

- 表结构创建和升级交给 Flyway；
- `la_schema_version` 可以逐步废弃或仅保留历史兼容；
- Flyway 使用自己的 `flyway_schema_history` 表记录迁移历史；
- `DatabaseInitializer` 保留为“默认管理员等基础数据初始化”，不再执行 schema SQL。

## 4. 技术可行性结论

结论：**可行，建议作为生产可靠性增强优先实施。**

可行原因：

- 项目已经完成数据库化，默认 H2，生产可切换 MySQL / PostgreSQL / GaussDB / OceanBase，具备持久化队列基础。
- 当前 Topic、订阅、通知目标、审计日志都已在数据库中，新增投递任务表符合现有架构方向。
- 现有通知通道已封装完成，可以复用邮件、Webhook、钉钉、企业微信、飞书等发送实现。
- Flyway 是成熟数据库迁移工具，适合解决“新版本表结构自动升级”的问题。
- 不引入外部 MQ，符合 Lite-Alert “单 JAR + 默认 H2 开箱即用”的产品定位。

需要同步解决的关键问题：

1. Flyway 多数据库迁移脚本组织；
2. 旧 `DatabaseInitializer` 与 Flyway 的职责拆分；
3. Webhook 成功响应码从 202 改为 200；
4. 投递任务落库和重启恢复；
5. 多实例并发领取；
6. payload 完整存储与权限化查询；
7. 投递任务保留周期配置化和即时生效；
8. 审计和统计口径兼容。

## 5. 推荐总体方案

推荐方案：

```text
数据库持久化投递队列 + Flyway 迁移 + 后台 Worker 扫描投递 + payload 权限化查询
```

不使用 Kafka / RabbitMQ / Redis。

架构链路：

```text
Webhook 请求
  → 参数校验 / 鉴权 / 限流 / Schema 校验
  → 开启事务
  → 写 webhook.accepted 审计
  → 按订阅目标创建 la_notify_delivery 投递任务
  → 提交事务
  → 返回 200 OK

后台 Worker
  → 定时扫描到期任务
  → 条件更新领取任务
  → 调用通知通道发送
  → 成功：SENT + notify.sent
  → 失败可重试：RETRY_WAIT + notify.failed
  → 达最大次数：GIVE_UP + notify.give_up

系统重启 / 新版本部署
  → Flyway 自动执行缺失 migration
  → 应用启动后恢复超时 SENDING 任务
  → 继续扫描 PENDING / RETRY_WAIT 任务
```

## 6. 响应码方案：成功返回 200

### 6.1 为什么不用 202

HTTP 语义上，202 表示“已接受但尚未处理完成”，适合异步任务。但实际对接中，很多第三方系统或低代码平台只把 200 当成功，202 可能被识别为失败。

因此本项目按产品兼容性优先：

```text
成功接收并落库后返回 200 OK
```

### 6.2 200 的业务含义

这里的 200 不表示“通知已经全部送达”，而表示：

```text
Lite-Alert 已成功接收请求，并已把后续投递任务可靠保存到数据库。
```

建议响应体只返回摘要，不返回每个投递任务明细：

```json
{
  "accepted": true,
  "traceId": "...",
  "deliveryCount": 2
}
```

如果无订阅目标：

```json
{
  "accepted": true,
  "traceId": "...",
  "deliveryCount": 0,
  "message": "no subscribers"
}
```

如果校验失败、鉴权失败、限流、payload 超限等，仍返回对应错误码：

| 场景 | HTTP 状态 |
|------|-----------|
| 鉴权失败 | 401 / 403 |
| Topic 不存在 | 404 |
| 限流 | 429 |
| payload 过大 | 413 |
| schema 校验失败 | 400 |
| 投递任务落库失败 | 500 |

## 7. 数据库表设计

### 7.1 新增投递任务表

新增表：`la_notify_delivery`

字段建议：

| 字段 | 类型建议 | 必填 | 含义 |
|------|----------|------|------|
| `id` | varchar(64) | 是 | 投递任务 ID |
| `trace_id` | varchar(128) | 否 | Webhook 请求追踪 ID |
| `topic_id` | varchar(64) | 是 | Topic ID |
| `target_id` | varchar(64) | 是 | 通知目标 ID |
| `channel` | varchar(32) | 是 | EMAIL / WEBHOOK / DINGTALK / WECOM / FEISHU |
| `payload_json` | clob / text / longtext | 是 | 完整原始消息体，用于恢复投递和授权查询 |
| `status` | varchar(32) | 是 | PENDING / SENDING / RETRY_WAIT / SENT / GIVE_UP / CANCELLED |
| `attempt` | int | 是 | 已尝试次数，默认 0 |
| `max_attempts` | int | 是 | 最大尝试次数，默认 5 |
| `next_retry_at` | timestamp | 是 | 下次可执行时间 |
| `locked_by` | varchar(128) | 否 | 当前领取任务的实例标识 |
| `locked_at` | timestamp | 否 | 领取时间 |
| `last_error` | clob / text / longtext | 否 | 最近一次失败原因，脱敏和截断后保存 |
| `created_at` | timestamp | 是 | 创建时间 |
| `updated_at` | timestamp | 是 | 更新时间 |
| `finished_at` | timestamp | 否 | 终态时间 |

状态说明：

| 状态 | 含义 | 是否终态 |
|------|------|----------|
| `PENDING` | 已创建，等待首次发送 | 否 |
| `SENDING` | 已被某个 Worker 领取，正在发送 | 否 |
| `RETRY_WAIT` | 发送失败，等待下次重试 | 否 |
| `SENT` | 发送成功 | 是 |
| `GIVE_UP` | 达到最大重试次数，最终放弃 | 是 |
| `CANCELLED` | 目标被删除或禁用等原因取消投递 | 是 |

推荐索引：

```text
idx_notify_delivery_due(status, next_retry_at)
idx_notify_delivery_lock(status, locked_at)
idx_notify_delivery_trace(trace_id)
idx_notify_delivery_topic(topic_id)
idx_notify_delivery_target(target_id)
idx_notify_delivery_created(created_at)
idx_notify_delivery_finished(finished_at)
```

### 7.2 新增用户权限字段或权限表

当前用户模型只有：

```text
ADMIN / USER
```

为了支持“管理员授权用户查看完整 payload”，需要新增权限模型。

推荐最小方案：在 `la_user` 增加 `permissions_json` 字段。

示例权限：

```json
[
  "DELIVERY_PAYLOAD_READ"
]
```

权限含义：

| 权限 | 含义 |
|------|------|
| `DELIVERY_PAYLOAD_READ` | 可以查看完整投递 payload |

规则：

- ADMIN 默认拥有该权限；
- USER 默认没有；
- 管理员可以在用户管理页勾选授权；
- 后端接口统一判断 `currentUser.isAdmin()` 或用户权限包含 `DELIVERY_PAYLOAD_READ`。

后续如果权限项增多，可再拆为独立 `la_user_permission` 表。本期建议 `permissions_json` 足够轻量。

### 7.3 系统设置新增投递保留周期

当前 `SystemSettings` 已有：

```text
auditRetention
```

新增：

```text
deliveryRetention
```

默认值建议：

```text
90 DAYS
```

系统设置页新增配置项：

```text
投递记录保留：90 天 / N 月 / N 年
```

修改后立即生效：

- 设置保存到 `la_system_settings`；
- `NotifyDeliveryJanitor` 每次执行时实时读取 `SystemSettingsService.current()`；
- 不缓存 retention；
- 不需要重启。

## 8. Flyway 数据库升级方案

### 8.1 引入 Flyway

后端新增依赖：

```xml
<dependency>
  <groupId>org.flywaydb</groupId>
  <artifactId>flyway-core</artifactId>
</dependency>
```

MySQL 如 Spring Boot 版本需要，可增加：

```xml
<dependency>
  <groupId>org.flywaydb</groupId>
  <artifactId>flyway-mysql</artifactId>
</dependency>
```

PostgreSQL 如需要，可增加：

```xml
<dependency>
  <groupId>org.flywaydb</groupId>
  <artifactId>flyway-database-postgresql</artifactId>
</dependency>
```

具体依赖以 Spring Boot 3.5 / Flyway 版本兼容性为准。

### 8.2 迁移脚本目录

推荐按数据库类型拆分脚本目录：

```text
backend/src/main/resources/db/migration/h2
backend/src/main/resources/db/migration/mysql
backend/src/main/resources/db/migration/postgresql
backend/src/main/resources/db/migration/gaussdb
backend/src/main/resources/db/migration/oceanbase
```

脚本示例：

```text
V1__baseline.sql
V2__notify_delivery.sql
V3__delivery_retention_and_payload_permission.sql
```

### 8.3 Flyway location 按数据库类型选择

在配置中根据 `lite-alert.database.type` 或 datasource 类型选择：

```text
spring.flyway.locations=classpath:db/migration/{type}
```

示例：

| 数据库 | Flyway location |
|--------|-----------------|
| H2 | `classpath:db/migration/h2` |
| MySQL | `classpath:db/migration/mysql` |
| PostgreSQL | `classpath:db/migration/postgresql` |
| GaussDB | `classpath:db/migration/gaussdb` |
| OceanBase | `classpath:db/migration/oceanbase` |

### 8.4 现有数据库如何接入 Flyway

因为当前项目已有 `schema-*.sql` 和 `la_schema_version`，引入 Flyway 时要处理两类情况：

#### 新安装

- Flyway 从 V1 开始执行；
- V1 创建当前已有所有基础表；
- V2 创建 `la_notify_delivery`；
- V3 增加 `permissions_json`、系统设置结构兼容等。

#### 老版本升级

老版本数据库已经有业务表，但没有 `flyway_schema_history`。

推荐处理方式：

- 启用 Flyway baseline；
- 对已有数据库设置 baseline version 为 1；
- 再执行 V2 / V3 等后续迁移。

配置方向：

```yaml
spring:
  flyway:
    enabled: true
    baseline-on-migrate: true
    baseline-version: 1
```

注意：

- V1 代表当前已存在的数据库结构基线；
- V2 开始放新增表和新增字段；
- `DatabaseInitializer` 不再执行 schema 文件，避免和 Flyway 重复建表。

### 8.5 DatabaseInitializer 调整

调整后职责：

| 组件 | 职责 |
|------|------|
| Flyway | 创建表、升级表结构、创建索引 |
| DatabaseInitializer | 初始化默认管理员等基础数据 |

`DatabaseInitializer` 应删除或停用：

```text
ScriptUtils.executeSqlScript(c, schema)
```

避免 Flyway 已经创建表后它再次执行 schema。

### 8.6 表结构变更能否自动识别

引入 Flyway 后答案是：**可以**。

新版本部署时：

1. Flyway 查看 `flyway_schema_history`；
2. 判断当前数据库执行到哪个版本；
3. 自动执行还没执行过的迁移脚本；
4. 成功后记录迁移历史；
5. 迁移失败则应用启动失败，避免旧表结构运行新代码。

## 9. Webhook 接收改造

### 9.1 当前逻辑

```text
校验通过 → audit webhook.accepted → dispatchAsync 内存异步派发 → 返回 202
```

### 9.2 改造后逻辑

```text
校验通过
  → 开启数据库事务
  → 写 webhook.accepted 审计
  → 根据订阅关系展开通知目标
  → 为每个启用目标写入 la_notify_delivery(PENDING)
  → 提交事务
  → 返回 200 OK
```

关键要求：

1. **返回 200 前必须完成投递任务落库**。
2. `webhook.accepted` 和投递任务创建建议在同一个事务内完成。
3. 如果没有订阅目标，写 `notify.no_subscribers` 审计，不创建投递任务，仍返回 200。
4. 如果某个订阅目标不存在或已禁用：
   - 跳过并写 `notify.cancelled` 或 `notify.target_unavailable` 审计；
   - 不创建无效投递任务。
5. 如果投递任务落库失败，不返回 200，应返回 500，让调用方知道本次未被可靠接收。

## 10. Worker 投递与锁定机制

### 10.1 Worker 扫描策略

新增后台 Worker，例如：`NotifyDeliveryWorker`。

推荐配置：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 扫描周期 | 5 秒 | 扫描到期任务 |
| 每批数量 | 50 | 避免单次拉取过多 |
| SENDING 超时 | 10 分钟 | 超过视为卡死，可恢复 |
| 最大尝试次数 | 5 | 与现有策略一致 |

扫描条件：

```text
status in ('PENDING', 'RETRY_WAIT')
and next_retry_at <= now
```

### 10.2 多实例安全领取

不依赖 `select for update skip locked`，避免多数据库兼容问题。

采用“先查候选，再条件更新抢占”的方式：

```sql
update la_notify_delivery
set status = 'SENDING', locked_by = ?, locked_at = ?, updated_at = ?
where id = ?
  and status in ('PENDING', 'RETRY_WAIT')
  and next_retry_at <= ?
```

如果影响行数为 1，说明本实例领取成功；如果为 0，说明已被其他实例领取或任务状态变化，跳过。

### 10.3 SENDING 卡死恢复

系统可能在任务状态已改为 `SENDING` 后宕机。为避免任务永远卡住，Worker 启动和定时扫描时执行恢复：

```text
status = SENDING
and locked_at < now - 10 分钟
```

恢复规则：

- 如果 `attempt < max_attempts`：改回 `RETRY_WAIT`，`next_retry_at = now`；
- 如果 `attempt >= max_attempts`：改为 `GIVE_UP`。

按当前产品确认，本期不重点处理“外部发送成功但未更新 SENT 就宕机导致重复”的极端场景。

## 11. 发送、重试和终态规则

### 11.1 成功

发送成功后：

```text
status = SENT
finished_at = now
last_error = null
```

写审计：

```text
notify.sent
```

审计 attrs 至少包含：

- `deliveryId`
- `traceId`
- `topicId`
- `targetId`
- `channel`
- `attempt`

### 11.2 失败但可重试

发送失败后，如果还没达到最大次数：

```text
attempt = attempt + 1
status = RETRY_WAIT
next_retry_at = now + backoff(attempt)
last_error = sanitized(error)
locked_by = null
locked_at = null
```

写审计：

```text
notify.failed
```

### 11.3 最终放弃

如果达到最大次数：

```text
status = GIVE_UP
finished_at = now
last_error = sanitized(error)
```

写审计：

```text
notify.give_up
```

### 11.4 重试节奏

保留现有节奏：

```text
1m → 5m → 30m → 2h → 6h
```

默认最大尝试次数：5。

### 11.5 目标被删除或禁用

Worker 执行时如果发现通知目标已不存在或被禁用：

```text
status = CANCELLED
finished_at = now
last_error = target unavailable or disabled
```

建议写审计：

```text
notify.cancelled
```

## 12. payload 完整记录与权限化查询

### 12.1 存储原则

投递任务必须完整保存原始 payload：

```text
la_notify_delivery.payload_json
```

用途：

1. 系统重启后恢复投递；
2. 管理员或授权用户排查问题；
3. 支撑后续投递详情页面。

### 12.2 查询权限

新增权限：

```text
DELIVERY_PAYLOAD_READ
```

权限规则：

| 用户 | 返回 payload |
|------|--------------|
| 拥有 `DELIVERY_PAYLOAD_READ` | 完整 payload |
| 无 `DELIVERY_PAYLOAD_READ` | 脱敏 payload |

### 12.3 脱敏规则

无权限用户查询时，后端返回脱敏后的 payload。

建议默认脱敏字段名包含：

```text
password
passwd
pwd
token
secret
authorization
apikey
api_key
access_key
private_key
credential
```

脱敏表现：

```json
{
  "username": "demo",
  "password": "***",
  "token": "***"
}
```

对于嵌套 JSON，需要递归脱敏。

对于非 JSON 文本或无法解析结构：

- 有权限用户返回原文；
- 无权限用户返回固定提示或截断脱敏文本，例如：`***masked***`。

### 12.4 授权入口

用户管理页新增权限配置：

```text
允许查看完整投递 Payload
```

后端用户更新接口支持保存 `permissions_json`。

审计记录：

- 管理员给用户授权或取消授权时，写 `user.update` 审计；
- attrs 中记录权限变化，但不记录 payload 内容。

## 13. 投递记录保留周期配置

### 13.1 系统设置新增项

在系统设置中新增：

```text
deliveryRetention
```

默认值：

```text
90 DAYS
```

前端系统设置页新增：

```text
投递记录保留
```

说明文案建议：

```text
超过保留期的已完成投递记录会自动从数据库清理，未完成任务不会清理。修改后立即生效。
```

### 13.2 立即生效机制

`NotifyDeliveryJanitor` 每次执行时实时读取：

```java
settings.current().getDeliveryRetention()
```

不做长期缓存。

保存系统设置后：

- 新配置立即写入 `la_system_settings`；
- 下一次清理任务执行时使用新配置；
- 如需要“保存后马上清理”，可以在保存接口中触发一次异步 janitor sweep，但不是必须。

### 13.3 清理范围

只清理终态任务：

```text
SENT / GIVE_UP / CANCELLED
```

不清理：

```text
PENDING / SENDING / RETRY_WAIT
```

避免未完成任务被误删。

## 14. 风险点闭环方案

### 14.1 宕机丢消息

解决方案：

- 请求返回 200 前，投递任务必须写入 `la_notify_delivery`；
- Worker 只处理数据库任务；
- 系统重启后扫描 `PENDING / RETRY_WAIT / 超时 SENDING` 继续处理；
- 停用内存 `RetryQueue` 作为主链路。

结果：**已接收并落库的消息不会因应用重启而丢失。**

### 14.2 调用方不识别 202

解决方案：

- 成功接收统一返回 200；
- 响应体中明确 `accepted: true` 和投递任务数量；
- 错误场景仍返回对应错误状态码。

结果：**兼容只认 200 的外部系统。**

### 14.3 多实例并发抢任务

解决方案：

- 使用 `locked_by / locked_at` 字段；
- 通过条件更新领取任务；
- 只有更新成功的实例可以发送；
- 不依赖特定数据库锁语法。

结果：**兼容多数据库，同时支持未来多实例部署。**

### 14.4 SENDING 任务永久卡住

解决方案：

- Worker 启动和定时任务中扫描超时 `SENDING`；
- 超过 10 分钟自动恢复为 `RETRY_WAIT`；
- 超过最大尝试次数则 `GIVE_UP`。

结果：**任务不会永久卡死。**

### 14.5 数据库表无限增长

解决方案：

- 加索引；
- Worker 分批处理，默认每批 50；
- 新增清理任务；
- 系统设置页配置 `deliveryRetention`；
- 修改后下一次清理即时使用新配置；
- 只清理终态任务。

结果：**数据库增长可控，且保留周期由产品/运维可配置。**

### 14.6 完整 payload 安全查询

解决方案：

- 数据库完整保存 payload；
- 普通应用日志不打印 payload；
- 查询接口按权限返回完整或脱敏 payload；
- 管理员可授权用户 `DELIVERY_PAYLOAD_READ`；
- 权限变更写审计。

结果：**兼顾排查能力和普通用户数据保护。**

### 14.7 多数据库升级

解决方案：

- 引入 Flyway；
- 按数据库类型维护 migration；
- 老库 baseline 到 V1；
- 新版本自动执行 V2/V3 等升级脚本；
- 迁移失败则应用启动失败。

结果：**新版本部署可以自动识别并升级表结构。**

### 14.8 审计和仪表盘兼容

解决方案：

- 继续写现有审计事件类型；
- 在 attrs 中补充 `deliveryId`；
- 统计口径保持不变；
- 后续可新增投递状态统计。

结果：**前端现有审计和趋势统计不需要大改。**

## 16. 测试与验证方案

后端测试必须覆盖：

1. **Flyway 迁移**
   - 空数据库能执行 V1/V2/V3 到最新版本；
   - 已有老库 baseline 后能自动执行后续 migration；
   - 重复启动不会重复执行 migration；
   - migration 失败时应用启动失败。

2. **响应码**
   - Webhook 校验通过并落库成功后返回 200；
   - 投递任务落库失败返回 500；
   - 鉴权、限流、schema 错误仍返回对应错误码。

3. **任务创建**
   - Webhook accepted 后，为每个启用订阅目标创建 `PENDING` 任务；
   - 无订阅目标时不创建任务，写 `notify.no_subscribers`。

4. **任务领取**
   - 到期任务能被 Worker 领取为 `SENDING`；
   - 条件更新失败时不会重复领取；
   - 模拟两个 Worker 抢同一任务，只有一个成功。

5. **发送成功**
   - 发送成功后状态为 `SENT`；
   - 写 `notify.sent`；
   - 记录 `deliveryId`。

6. **发送失败和重试**
   - 第一次失败后状态为 `RETRY_WAIT`；
   - `attempt` 增加；
   - `next_retry_at` 正确设置；
   - 写 `notify.failed`。

7. **最终失败**
   - 达到最大次数后状态为 `GIVE_UP`；
   - 写 `notify.give_up`。

8. **宕机恢复模拟**
   - 构造超时 `SENDING` 任务；
   - Worker 启动后恢复为 `RETRY_WAIT` 或 `GIVE_UP`。

9. **保留周期即时生效**
   - 修改系统设置 `deliveryRetention`；
   - 不重启应用；
   - 下一次 janitor 执行使用新保留周期；
   - 未完成任务不被清理。

10. **payload 权限**
   - ADMIN 查询完整 payload；
   - 拥有 `DELIVERY_PAYLOAD_READ` 的 USER 查询完整 payload；
   - 无权限 USER 查询脱敏 payload；
   - 嵌套敏感字段递归脱敏。

11. **普通日志安全**
   - 发送失败日志不打印完整 payload；
   - last_error 截断和脱敏。

## 19. 推荐结论

推荐采用：

```text
数据库持久化投递队列 + Flyway 数据库迁移 + 200 成功响应 + payload 权限化查询 + 可配置投递保留周期
```

明确不使用 Kafka / RabbitMQ / Redis。

最终产品口径：

1. Webhook 请求校验通过且投递任务落库成功后返回 200。
2. 返回 200 表示 Lite-Alert 已可靠接收并保存投递任务，不表示外部通道已经送达。
3. 系统重启后会继续处理未完成投递任务。
4. 投递记录保留周期可在系统设置页配置，修改后立即对后续清理生效。
5. 完整 payload 会保存；管理员可授权用户查看完整 payload；无权限用户只能查看脱敏 payload。
6. 新版本部署时，Flyway 自动识别并升级数据库表结构。

确认后建议进入阶段一开发。

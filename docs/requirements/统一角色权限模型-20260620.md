# 统一角色权限模型

**日期**：2026-06-20
**提出人**：产品经理
**需求类型**：权限模型优化

## 1. 需求概述

当前用户创建页面同时存在两类概念：

- `ADMIN / USER` 角色 radio；
- 可配置角色绑定。

产品期望去掉 `ADMIN / USER` 这套固定账号类型，不再让系统中存在两套角色概念。后续统一采用：

> 用户绑定角色，角色包含权限，权限决定用户能访问和操作哪些功能。

初始 `admin` 用户不再依赖 `ADMIN` 字段获得管理员能力，而是默认绑定一个内置的「超级管理员」角色；该角色拥有全部权限。

## 2. 技术可行性结论

可行，但属于权限模型重构，不是单纯删除前端 radio。

原因是当前系统中 `ADMIN / USER` 不只是页面字段，还被后端鉴权、前端路由、JWT、数据初始化和部分业务可见性判断依赖。直接删除会导致用户管理、系统设置、全局统计、命名空间/Topic/ApiKey 可见性等功能出现权限异常。

推荐按一次完整改造处理：

1. 产品层移除 `ADMIN / USER` 展示；
2. 后端鉴权从 `ROLE_ADMIN` 改为细粒度权限；
3. 初始管理员默认绑定内置「超级管理员」角色；
4. 用户创建只提交角色绑定；
5. 前端菜单、路由、页面可见性全部改用权限判断；
6. 数据库保留或废弃 `la_user.role` 字段需谨慎处理，避免破坏已有数据库升级。

## 3. 推荐方案及理由

### 3.1 权限模型

统一为：

```text
User 用户
  -> UserRole 用户-角色绑定
      -> Role 角色
          -> Permission 权限
```

不再使用 `ADMIN / USER` 作为产品或鉴权依据。

### 3.2 内置角色

保留并明确内置角色：

| 角色 ID | 角色名称 | 说明 |
| --- | --- | --- |
| `r_super_admin` | 超级管理员 | 系统内置角色，拥有全部权限，不允许删除 |
| `r_normal_user` | 普通用户 | 系统内置角色，默认无管理权限，可作为普通账号默认角色 |

初始 `admin` 用户默认绑定 `r_super_admin`。

### 3.3 后端改造方向

#### 用户模型

- 用户创建接口不再接收 `role` 字段。
- 用户更新接口不再更新 `role` 字段。
- 用户返回给前端的数据不再暴露 `role`。
- 用户权限统一由绑定角色计算。

#### 鉴权逻辑

把当前依赖 `ADMIN` 的判断迁移为权限判断：

| 当前逻辑 | 调整后逻辑 |
| --- | --- |
| `/api/admin/**`、`/api/users/**` 依赖 `ROLE_ADMIN` | 根据具体接口检查 `SYSTEM_SETTINGS_*`、`USER_*`、统计权限等 |
| `currentUser.isAdmin()` 判断能否看全部数据 | 改为判断是否拥有对应的全局查看权限 |
| JWT authority 来自 `User.Role` | JWT / SecurityContext authority 来自用户实际权限集合 |

#### 权限补充

当前权限中已有大部分管理权限，但“查看全部资源”类能力需要补齐或明确：

- `NAMESPACE_VIEW_ALL`：查看全部命名空间；
- `TOPIC_VIEW_ALL`：查看全部 Topic；
- `APIKEY_VIEW_ALL`：查看全部 ApiKey；
- `CONTACT_VIEW_ALL`：查看全部通知目标；
- `STATS_VIEW`：查看系统统计/全局仪表盘。

如果不新增这些权限，也可以复用已有 `NAMESPACE_VIEW`、`TOPIC_VIEW` 等权限表示全局可见，但语义不够清晰。推荐新增 `*_VIEW_ALL`，避免普通查看权限和跨用户查看权限混淆。

### 3.4 前端改造方向

用户管理页面：

- 删除创建用户弹窗里的 `ADMIN / USER` radio；
- 删除用户列表中的固定 `ADMIN / USER` 标签；
- 「角色绑定」改名为「角色」或「权限角色」；
- 创建用户时只提交 `username`、`password`、`roleIds`。

权限判断：

- 删除 `auth.isAdmin` 对页面和菜单的控制；
- `auth.hasPermission()` 不再因为 `role === 'ADMIN'` 直接放行；
- 路由 `adminOnly` 改成具体 `meta.permission`；
- 仪表盘全局统计、系统设置、用户管理等页面使用具体权限控制。

### 3.5 数据库与兼容策略

项目支持 H2、MySQL、PostgreSQL、GaussDB、OceanBase，且已有 migration。直接删除 `la_user.role` 字段会涉及多数据库 DDL 差异，风险较高。

推荐本轮做“业务废弃”：

- 不再读写 `role` 作为权限依据；
- 新建用户不再由前端传 `role`；
- 后端如数据库字段仍要求非空，可内部写入固定兼容值，例如 `USER`；
- API 响应不再暴露 `role`；
- 后续单独做数据库清理 migration，再物理删除 `role` 字段。

这样可以实现产品上的“不保留”，同时降低多数据库升级风险。

## 5. 风险点和注意事项

1. **权限丢失风险**
   - 如果初始 admin 没有正确绑定超级管理员角色，会导致无法进入后台管理。

2. **接口放开或误拒风险**
   - 需要逐个替换 `ROLE_ADMIN` 和 `isAdmin()`，否则可能出现有权限的人进不去、没权限的人进得去。

3. **多数据库迁移风险**
   - 不建议本轮直接物理删除 `la_user.role` 字段。
   - 本轮先业务废弃，后续再清理数据库字段。

4. **前后端字段契约变化**
   - 前端类型、用户列表、登录用户信息都要同步调整，避免继续读取 `role`。

5. **权限命名需要统一**
   - 是否新增 `*_VIEW_ALL` 和 `STATS_VIEW` 需要在开发前确认。推荐新增，语义更清楚。

## 6. 预估工作量

- 后端权限模型与接口调整：1.5 ~ 2 天；
- 前端用户管理、路由、菜单和页面权限调整：0.5 ~ 1 天；
- 测试补充与回归验证：1 天；
- 总计：约 3 ~ 4 人日。

## 7. 推荐实施顺序

1. 补充测试，覆盖“绑定超级管理员角色即可获得管理能力”；
2. 新增/明确全局查看和统计权限；
3. 后端 SecurityConfig 与 CurrentUser 从基础角色迁移到权限；
4. 用户创建/更新接口移除 `role` 参数依赖；
5. 初始 admin 和历史 ADMIN 用户自动绑定超级管理员角色；
6. 前端删除 `ADMIN / USER` 展示和判断；
7. 前端用户创建只选择角色；
8. 运行后端测试、前端 type-check 和 build；
9. 更新设计文档中的认证权限说明。

## 8. 待产品确认项

推荐默认采用以下决策：

1. 产品层彻底不展示 `ADMIN / USER`；
2. 本轮不物理删除数据库 `la_user.role` 字段，只做业务废弃；
3. 新增 `*_VIEW_ALL` 与 `STATS_VIEW` 权限，用于替代历史管理员全局可见能力；
4. 初始 admin 用户默认绑定内置「超级管理员」角色；
5. 后续创建用户只选择角色，不选择账号类型。
